# ADR-003: 開発イテレーションはVMサービス経由のホットリスタートで再実行する

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: [ADR-001](./01-event-driven-waiting.md)、[ADR-002](./02-agent-transport.md)

## 背景・課題

「1修正=1フルビルド」はE2Eテスト開発最大の痛点である。`endevir develop` がホットリスタートによる再実行でこれを解消できるか、目標10秒以内としてループ時間を計測した。

## 検証結果（iPhone 17 Proシミュレータ / debugビルド）

構成: 起動時にミニテストシナリオ（ホーム→画面遷移タップ→カウントアップタップ→検証）を自動実行するエントリポイント + `flutter attach` へのstdinキー送信（R=ホットリスタート）。コード到達はバージョン印の書き換えで確認。

| 計測                                            | 結果                                                            |
| ----------------------------------------------- | --------------------------------------------------------------- |
| **修正→ホットリスタート→シナリオ再実行完了**    | **1.75秒 / 2.00秒 / 2.22秒**（3回、コード変更の到達を毎回確認） |
| ホットリスタート単体                            | 1.0〜2.2秒                                                      |
| シナリオ実行（待機・actionabilityチェック込み） | 1.19〜1.35秒                                                    |
| 目標（10秒以内）                                | **大幅達成**                                                    |

## 決定

- `endevir develop` はdebugビルド+VMサービス経由のホットリスタートで「修正→再実行」ループを実現する（Patrol方式の有効性を追認）
- `flutter attach` のstdinキー転送には依存せず、**VM Service APIを直接呼び出す**（アタッチ11秒の同期を省略でき、mDNS曖昧性も回避）

## 実装上の知見

1. **exists ≠ actionable（最重要）**: 画面遷移アニメーションの初フレームで遷移先のテキストは既にツリーに存在するため、存在確認だけで即タップすると**遷移中の要素を空振りする**（初回計測は3回全滅）。要素単位の**位置安定チェック**（連続3フレームでグローバル座標が不変）を挟むと3/3成功。全体静止待ち（quiescence）は無限アニメーション画面で使えないため、**per-element actionability checkはタップの必須前段**とする
2. **flutter attachのmDNS曖昧性**: 過去に起動した同一Bundle IDのVM Service広告が残り「more than one Dart VM Service report」で誤った対象を掴みうる。本番は起動時にVM Service URIを直接取得して接続する（--debug-url相当）
3. **attach経由のアプリログ転送は信頼できない**（今回の構成では1行も届かず、unified log側で確認）。テストログ・結果の転送は**自前のWebSocketチャネル（ADR-002）で行い、attachのstdoutには依存しない**

## 影響

- `endevir develop` のアーキテクチャは「debugビルド起動 → VM Service直接接続 → ファイル監視 → ホットリスタート → エージェント（ADR-002）経由で結果受信」とする
- actionability check（位置安定）はタップAPIの既定動作とし、安定判定フレーム数を設定可能にする
- ホットリスタートはFlutterレイヤーのみを対象とし、ネイティブ側の変更時はフルビルドする
- 見直しトリガー: VM Service APIの直接呼び出しがattach経由より不安定だった場合はattach方式に戻す

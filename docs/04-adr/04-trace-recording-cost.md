# ADR-004: 証跡のスクリーンショットはGPUスナップショットのみ同期、エンコードは遅延する

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: RPT-001/002/004/006、S4スパイク、[検証コード](../../examples/flutter_app/lib/main_s4_trace.dart)、[[01-event-driven-waiting]]、[[03-hot-restart-loop]]

## 背景・課題

証跡モード（RPT-004: 成功時も全ステップの手順+スクリーンショットを記録）はプロダクト仮説の核だが、記録オーバーヘッドがテスト実行を著しく劣化させては本末転倒（目標: +10%未満）。ステップ相関trace（RPT-001/002）とスクリーンショットの実コストをM0で計測した。あわせて言語中立スキーマの型生成ツールチェーン（RPT-006）を確認した。

## 検証結果（S4スパイク、iPhone 17 Proシミュレータ / debugビルド、4ステップ×10反復）

| モード                                            | 反復あたり | ベースライン比         |
| ------------------------------------------------- | ---------- | ---------------------- |
| A: 記録なし（ベースライン）                       | 1351ms     | —                      |
| B: trace（ステップ+タイムスタンプ+JSONL書き込み） | 1326ms     | **-1.9%（実質ゼロ）**  |
| C: 証跡モード・同期スクショ（毎ステップPNG保存）  | 1665ms     | **+23.3%（目標超過）** |
| D: 証跡モード・遅延エンコード                     | 1327ms     | **-1.8%（実質ゼロ）**  |

スクリーンショット1枚（60KB PNG、論理解像度）の内訳:

| 工程                                         | 時間       | 備考                             |
| -------------------------------------------- | ---------- | -------------------------------- |
| GPUスナップショット（`OffsetLayer.toImage`） | **1ms**    | この時点でフレーム内容が確定する |
| PNGエンコード（`toByteData(png)`）           | 121〜145ms | コストのほぼ全て                 |
| ファイル書き込み                             | 1ms        |                                  |

## 決定

1. **trace記録（ステップ+ログ相関+JSONL）は常時オンにできる**（オーバーヘッド実質ゼロ）。記録モードのプリセット（RPT-004）はスクリーンショット量の制御であり、ステップ記録自体を落とす必要はない
2. **証跡モードのスクリーンショットは「同期はGPUスナップショットのみ・エンコード/書き込みは遅延キュー」方式にする**。フレーム内容はtoImage時点（1ms）で確定するため、遅延しても証跡の正しさは保たれる。これにより**証跡モードのオーバーヘッドは実質ゼロ**（目標+10%を大幅に下回る）
3. **trace/証跡スキーマの型生成はquicktype（JSON Schema → Dart / TypeScript）を採用**。Dart側はfromJson/toJson+enum、TS側はランタイム検証付きで生成されることを確認済み。スキーマ定義を単一の真実としてCIで両言語の型を生成する（RPT-006）

## 制約・注意（実装時）

- 遅延キューは**有界**にする（ui.Imageを保持するためメモリを消費する。上限超過時は同期フォールバックまたは間引き）。テスト終了時に`close()`で全エンコード完了を待つ
- `debugLayer`経由のキャプチャはdebugビルド限定。profile/releaseや本番ランナーでのキャプチャ機構（エンジンAPI / プラットフォーム別スクリーンショット）はM1で設計する（未解決）
- PNGエンコードはメインisolateのUIスレッド外（エンジン内部）で走るが、大量連続実行時のジャンク有無は実機で再計測する
- 計測はシミュレータ・debugビルドの値。実機/profileでの再計測をM6ベンチマークに含める

## 影響

- RPT-004の記録プリセット設計が単純化する: 「常時trace + スクショ量の選択（evidence / on-first-retry / only-on-failure）」
- 証跡モードを既定オンにする選択肢が現実味を持つ（マーケティング上も「証跡は無料でついてくる」と言える）
- スキーマ駆動開発のツールチェーンが確定し、M1のプロトコル/traceスキーマ設計に入れる
- 見直しトリガー: 実機計測でエンコードがフレームジャンクを起こす場合（対策候補: 解像度スケール、フォーマット変更、isolate分離）

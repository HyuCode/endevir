# ADR-005: テスト列挙は構文解析による静的抽出で行い、ドライランを廃する

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: CORE-110、S2スパイク、[列挙器](../../examples/flutter_app/tool/s2_enumerate.dart) / [検証対象](../../examples/flutter_app/endevir_test/)、[[02-agent-transport]]

## 背景・課題

ネイティブテスト写像（CORE-109）とテスト単位の選択実行には、実行前にテスト階層の一覧が必要になる。Patrolは `test_api` に静的な列挙APIがないため「アプリを起動して全テストを空実行するドライラン」で階層を取得しており、公式自身が「最もトリッキーな部分」と認める起動オーバーヘッドと複雑さの源泉になっている（02-patrol-analysis.md §2.3）。CORE-110はこれをビルド時の静的列挙で置き換えることを要求する。

## 検証結果（S2スパイク）

package:analyzerの**構文解析のみ**（意味解析なし・依存解決不要で高速）によるAST走査で検証。

| 検証項目                                                         | 結果                                                                                                      |
| ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `endevirTest` / `endevirGroup` の抽出（グループ階層含む）        | ✅ `ログイン > 正しい認証情報でログインできる` 形式で正確に抽出                                           |
| ファイル内const参照の文字列補間の解決                            | ✅ `'$_feature画面が表示される'` → `'ホーム画面が表示される'`                                             |
| 動的なテスト名（ループ変数の補間・ループ内宣言）の検出           | ✅ 静的解決不能として行番号+理由付きで警告                                                                |
| テストバンドル生成（静的テスト名→対象1件に絞った実行クロージャ） | ✅ 生成物が `flutter analyze` を通過                                                                      |
| 列挙速度                                                         | **60ファイル・300テストで103ms**（2ファイルでも約100ms=アナライザ初期化が支配的で、規模に対してほぼ定数） |

## 決定

1. **テスト列挙はビルド時の構文解析（AST）で行う**。実行時ドライランは行わない。生成物は (a) テストマニフェスト（JSON、ネイティブランナー側のテストケース生成に使用）、(b) テストバンドル（静的テスト名→実行クロージャのマップ）
2. **テスト名は静的に解決可能でなければならない**という規約を置く。解決可能な範囲: 文字列リテラル、隣接文字列、ファイル内const Stringの単純な補間。ループ・変数由来の動的テスト名は列挙器が行番号付きで警告し、`endevir doctor` / lintで開発時に指摘する（データ駆動テストは「静的な名前+パラメータ」のAPI形状を提供して吸収する。API設計で対応）
3. **`endevirTest` は「登録」であり実行はランナーが担う**（flutter_testと同じ構造）。テストファイルの `main()` は同期的に登録のみ行う。スパイクで「mainを直接awaitする」形が型エラーになることを確認済みで、登録/実行の分離が構造的に正しい

## 検討した選択肢（不採用）

- **実行時ドライラン（Patrol方式）**: 実績はあるが、起動オーバーヘッド・「Discoveryが最もトリッキー」という複雑さ・コールドスタートのフレーク源を引き継ぐ。不採用
- **意味解析（resolveFile）ベースの列挙**: 補間解決の精度は上がるが、依存解決を伴い数十秒オーダーになる。構文解析+規約で十分。不採用（規約で不足が出た場合の拡張候補として保持）

## 影響

- CORE-110の実装方式が確定。`endevir_cli` のビルドパイプラインに列挙器を組み込み、マニフェストとバンドルを生成する
- ネイティブ写像（S6/CORE-109）は「マニフェストからネイティブ側テストケースを生成し、各ケースがバンドルのクロージャをエージェント（ADR-002）経由で起動する」構造にできる。Patrolの「起動時にDart側へ階層を問い合わせる」同期が不要になる
- テスト名の静的解決規約はドキュメント・lintで開発者に明示する（暗黙の制約にしない）
- 見直しトリガー: 規約が実プロジェクトで受け入れられない場合（データ駆動テストの需要調査をα後に実施）

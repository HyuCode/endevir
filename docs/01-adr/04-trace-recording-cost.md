# ADR-004: 証跡のスクリーンショットはGPUスナップショットのみ同期、エンコードは遅延する

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: [ADR-001](./01-event-driven-waiting.md)、[ADR-003](./03-hot-restart-loop.md)

## 背景・課題

成功時も全ステップの手順とスクリーンショットを記録する証跡モードは、テスト実行を著しく遅くしてはならない（目標: +10%未満）。ステップ相関traceとスクリーンショットの実コストを計測し、言語中立スキーマの型生成ツールチェーンも確認した。

## 検証結果（iPhone 17 Proシミュレータ / debugビルド、4ステップ×10反復）

| モード                                            | 反復あたり | ベースライン比         |
| ------------------------------------------------- | ---------- | ---------------------- |
| A: 記録なし（ベースライン）                       | 1351ms     | —                      |
| B: trace（ステップ+タイムスタンプ+JSONL書き込み） | 1326ms     | **-1.9%（実質ゼロ）**  |
| C: 証跡モード・同期スクショ（毎ステップPNG保存）  | 1665ms     | **+23.3%（目標超過）** |
| D: 証跡モード・遅延エンコード                     | 1327ms     | **-1.8%（実質ゼロ）**  |

スクリーンショット1枚（60KB PNG、論理解像度）の内訳:

| 工程                                         | 時間       | 備考                             |
| -------------------------------------------- | ---------- | -------------------------------- |
| GPUスナップショット（`OffsetLayer.toImage`） | **1ms**    | この時点でフレーム内容が確定する |
| PNGエンコード（`toByteData(png)`）           | 121〜145ms | コストのほぼ全て                 |
| ファイル書き込み                             | 1ms        |                                  |

## 決定

1. **trace記録（ステップ+ログ相関+JSONL）は常時オンにできる**（オーバーヘッド実質ゼロ）。記録モードのプリセットはスクリーンショット量の制御であり、ステップ記録自体を落とす必要はない
2. **証跡モードのスクリーンショットは「同期はGPUスナップショットのみ・エンコード/書き込みは遅延キュー」方式にする**。フレーム内容はtoImage時点（1ms）で確定するため、遅延しても証跡の正しさは保たれる。これにより**証跡モードのオーバーヘッドは実質ゼロ**（目標+10%を大幅に下回る）
3. **trace/証跡スキーマの型生成はquicktype（JSON Schema → Dart / TypeScript）を採用**。Dart側はfromJson/toJson+enum、TS側はランタイム検証付きで生成されることを確認済み。スキーマ定義を単一の真実としてCIで両言語の型を生成する

## 制約・注意（実装時）

- 遅延キューは**有界**にする（ui.Imageを保持するためメモリを消費する。上限超過時は同期フォールバックまたは間引き）。テスト終了時に`close()`で全エンコード完了を待つ
- `debugLayer`経由のキャプチャはdebugビルド限定。profile/releaseや本番ランナーでは別のキャプチャ機構が必要
- PNGエンコードはメインisolateのUIスレッド外（エンジン内部）で走るが、大量連続実行時のジャンク有無は実機で再計測する
- 計測はシミュレータ・debugビルドの値であり、実機/profileでは再計測が必要

## 影響

- 記録プリセットは「常時trace + スクショ量の選択（evidence / on-first-retry / only-on-failure）」として単純化できる
- 証跡モードを既定オンにする選択肢が現実的になる
- プロトコルとtraceスキーマはスキーマ駆動で開発する
- 見直しトリガー: 実機計測でエンコードがフレームジャンクを起こす場合（対策候補: 解像度スケール、フォーマット変更、isolate分離）

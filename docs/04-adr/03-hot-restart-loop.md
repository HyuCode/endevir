# ADR-003: 開発イテレーションはVMサービス経由のホットリスタートで再実行する

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: CLI-102、NFR-002、S5スパイク、[検証コード](../../examples/flutter_app/lib/main_s5_hot_restart.dart)、[[01-event-driven-waiting]]、[[02-agent-transport]]

## 背景・課題

「1修正=1フルビルド」はE2Eテスト開発最大の痛点であり、`endevir develop`（CLI-102）はホットリスタートによる再実行でこれを解消する。Patrolの `patrol develop`（flutter attachにキー転送）と同方式が成立するか、ループ秒数（NFR-002のKPI、目標10秒以内）を計測した。

## 検証結果（S5スパイク、iPhone 17 Proシミュレータ / debugビルド）

構成: 起動時にミニテストシナリオ（ホーム→画面遷移タップ→カウントアップタップ→検証）を自動実行するエントリポイント + `flutter attach` へのstdinキー送信（R=ホットリスタート）。コード到達はバージョン印の書き換えで確認。

| 計測                                            | 結果                                                            |
| ----------------------------------------------- | --------------------------------------------------------------- |
| **修正→ホットリスタート→シナリオ再実行完了**    | **1.75秒 / 2.00秒 / 2.22秒**（3回、コード変更の到達を毎回確認） |
| ホットリスタート単体                            | 1.0〜2.2秒                                                      |
| シナリオ実行（待機・actionabilityチェック込み） | 1.19〜1.35秒                                                    |
| KPI（NFR-002: 10秒以内）                        | **大幅達成**                                                    |

## 決定

- `endevir develop` はdebugビルド+VMサービス経由のホットリスタートで「修正→再実行」ループを実現する（Patrol方式の有効性を追認）
- ただし `flutter attach` のstdinキー転送への依存は暫定とし、M1では**VM Service APIの直接呼び出し**（アタッチ11秒の同期を省略でき、mDNS曖昧性も回避）を第一候補としてプロトコル設計する

## スパイクで得た副次的発見

1. **exists ≠ actionable（最重要）**: 画面遷移アニメーションの初フレームで遷移先のテキストは既にツリーに存在するため、存在確認だけで即タップすると**遷移中の要素を空振りする**（初回計測は3回全滅）。要素単位の**位置安定チェック**（連続3フレームでグローバル座標が不変）を挟むと3/3成功。全体静止待ち（quiescence）は無限アニメーション画面で使えないため、**per-element actionability check（Playwrightのactionability checksに相当）はタップの必須前段**として CORE-102/CORE-004 の実装要件に昇格させる
2. **flutter attachのmDNS曖昧性**: 過去に起動した同一Bundle IDのVM Service広告が残り「more than one Dart VM Service report」で誤った対象を掴みうる。本番は起動時にVM Service URIを直接取得して接続する（--debug-url相当）
3. **attach経由のアプリログ転送は信頼できない**（今回の構成では1行も届かず、unified log側で確認）。テストログ・結果の転送は**自前のWebSocketチャネル（ADR-002）で行い、attachのstdoutには依存しない**

## 影響

- CLI-102の実装方式が確定。`endevir develop` のアーキテクチャは「debugビルド起動 → VM Service直接接続 → ファイル監視 → ホットリスタート → エージェント（ADR-002）経由で結果受信」
- actionability check（位置安定）はタップAPIの既定動作としてM1のコアAPI設計に組み込む（閾値: 安定判定フレーム数はCORE-103に従い設定可能にする）
- 制約（Patrolと同様）: ホットリスタートはFlutterレイヤーのみ。ネイティブ側の変更はフルビルドが必要。物理iOSデバイスでの可否はM1以降で検証
- 見直しトリガー: VM Service APIの直接呼び出しがattach経由より不安定だった場合はattach方式に戻す

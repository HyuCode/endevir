# ADR-002: エージェント通信は単一WebSocket常時接続を一次トランスポートにする

- 日付: 2026-07-17
- ステータス: Accepted
- 関連: CORE-105、S1スパイク、[エージェント](../../examples/flutter_app/lib/s1_agent.dart) / [ホストプローブ](../../examples/flutter_app/tool/s1_host_probe.dart)、[[01-event-driven-waiting]]

## 背景・課題

Endevirはアプリ内エージェント（テスト対象プロセス内のサーバ）とホスト（CLI/ランナー）の通信路を持つ。PatrolのCIフレーキーさの一因がHTTP同期であること（02-patrol-analysis.md §2.3）、Playwrightの安定性の基盤が単一常時接続であること（03-web-e2e-analysis.md §1.1）を踏まえ、トランスポートの成立性・レイテンシ・再接続耐性をM0で検証した。

## 検証結果（S1スパイク）

構成: アプリ内で `HttpServer.bind`（HTTP + WebSocketアップグレード）、論理座標のポインタイベント合成によるタップ、要素ツリー走査（キー列挙・テキスト検索）。**WidgetTester/flutter_testに依存しない本番モード動作**。

| 計測項目                                                           | iOSシミュレータ（直接localhost）       | Androidエミュレータ（adb forward）     |
| ------------------------------------------------------------------ | -------------------------------------- | -------------------------------------- |
| HTTP RTT（100回）                                                  | median 309µs / p95 655µs               | median 3.5ms / **p95 45ms・max 628ms** |
| WebSocket RTT（100回）                                             | median 113µs / p95 165µs               | median 1.4ms / **p95 2.8ms**           |
| 実操作シナリオ（画面遷移タップ→カウントアップタップ→テキスト検証） | 352ms                                  | 601ms                                  |
| ポインタ合成タップ                                                 | ✅ 成立                                | ✅ 成立                                |
| ツリーアクセス（キー列挙・テキスト検索）                           | ✅                                     | ✅                                     |
| kill→再起動→再接続                                                 | ✅（ホスト側リトライループで自動復帰） | 未計測（機構は同一）                   |

## 検討した選択肢

### 案A: WebSocket常時接続（1テストセッション=1接続）+ JSON-RPC

- レイテンシがHTTP比で2〜3倍速く、**Androidでのp95が桁違いに安定**（2.8ms vs 45ms。HTTPは接続確立コストが毎回乗る）
- サーバ→ホストの逆方向プッシュ（ログ・イベントストリーム、trace転送）が同一接続で可能。RPT-002のログ相関ストリームに必須
- Playwrightの実証済みパターン（単一コネクション+双方向Channel）と一致
- 採用

### 案B: HTTPリクエスト/レスポンス（Patrol型）

- 実装は最も単純だが、Androidでテールレイテンシが不安定。逆方向プッシュに別機構（ポーリング/SSE）が必要
- 不採用（ただしヘルスチェック `/ping` 等の補助エンドポイントとしては併用可）

## 決定

- **トランスポート**: WebSocket常時接続を一次とし、上にJSON-RPC 2.0を載せる（プロトコル分離の実体。全体編 §8.3）
- **到達経路**: iOSシミュレータ=直接localhost、Androidエミュレータ/実機=adb forward。iOS実機はUSB経由ポートフォワード（usbmuxd/iproxy系）をM1以降で検証
- **ポート**: 既定ポート+起動時ネゴシエーション（衝突時は空きポートに退避し、エージェントが標準出力/プラットフォームチャネルで通知）。詳細はM1のプロトコル設計で確定
- **再接続**: ホスト側リトライ（今回は500ms×30回で実証）を基本に、切断検知・診断ログ・冪等な再送をプロトコルに組み込む（CORE-105）

## 影響

- M1のプロトコル設計はJSON-RPC over WebSocketを前提に行う。メッセージスキーマはtrace/証跡スキーマ（RPT-006）と同じく言語中立に定義する
- ポインタ合成タップが本番モードで成立したため、**テストランナーなし（flutter run中のアプリへのアタッチ等）での操作**にも同じエージェントを使える見込み（record機能・MCPの土台）
- Android HTTPのテール不安定性の知見により、頻繁な小リクエストを伴う設計（ステップごとのHTTP往復）は禁止事項とする
- 見直しトリガー: iOS実機のポートフォワードが成立しない場合（代替: ネットワーク経由 or XCUITestランナー経由の中継）
